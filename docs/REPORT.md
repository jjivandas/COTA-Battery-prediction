## 1. What is this project

COTA operates a fleet of battery-electric transit buses in Central Ohio. Like every lithium-ion battery, the packs in those buses lose usable capacity over time — a slow, irreversible decay called *capacity loss* (Q-loss) or *State of Health (SoH) decline*. For a transit operator this matters a lot: as capacity drops, a bus's per-charge range drops, scheduling gets tighter, and at some point the pack needs to be replaced.

The Center for Automotive Research at OSU is collaborating with COTA to build a system that can predict that decline. The end-state we want is a tool that, given the operating history of a bus (driving, charging, temperatures, SOC behavior, etc.), forecasts how its battery's capacity will evolve — ultimately *live*, in real time, as the bus is driven.

We are not there yet. The first step — and the scope of this repository — is to build the modeling and evaluation infrastructure on **simulated** data generated by the CAR team. Real on-vehicle telemetry is expensive and lagging; simulation lets us:

- prototype the data-cleaning, feature-engineering, and modeling pipeline,
- decide what good evaluation looks like (especially: what does "predict the future" mean here?),
- understand which input variables genuinely drive predicted aging,
- and put a stake in the ground that says "this is what good performance looks like, and this is where we fall short."

Once the pipeline is solid on simulated data, the same code can be pointed at real data with minimal changes.

## 2. The data we are working with

### 2.1 Origin

All of the data used in this project is **simulated by the CAR team**. There are twelve bus simulations, named Bus-Id-001 through Bus-Id-012. Each bus has a single CSV containing weekly aggregates over the period roughly Jan 2020 – Oct 2023. The reference description that came with the drop (`data/raw/mar-2026/docs/00-Description of Data.pdf`) specifies 28 columns per file:

- **Time**: `Week_Index` (per-bus week counter, independent across buses), `Week_Start`, `Week_End`.
- **Operations**: `Routes` (comma-separated list of routes operated that week), `weeklyDistance`, `weeklyTotDistance` (cumulative).
- **Battery state-of-charge**: `Avg_SOC`, `Max_SOC`, `Min_SOC`, `Avg_DOD` (depth of discharge).
- **Temperatures**: `Avg_Batt_Temp`, `Avg_Amb_Temp`.
- **Power and energy**: `weeklyAvgBattP` (avg battery power), `weeklyDRegEn` and `weeklyTotRegEn` (regenerative energy weekly + cumulative), `weeklyDNetEn` and `weeklyTotNetEn` (net energy), `weeklyEnAux` and `weeklyTotAux` (auxiliary loads).
- **Charging**: `weeklyAvgCrateChg` (avg charge C-rate), `weeklyCycles` and `weeklyTotCycles` (equivalent full cycles).
- **Time budget**: `weeklyDrivingTime`, `weeklyChargingTime`, `weeklyparkingTime` (hours).
- **Aging — the thing we want to predict**:
  - `Avg_Qcyc` — *cycling* contribution to capacity loss (driven by use).
  - `Avg_Qcal` — *calendar* contribution to capacity loss (driven by time and temperature).
  - `Avg_Qloss = Avg_Qcyc + Avg_Qcal` — total capacity loss to date. This is our **target**.

Two important properties stated by the CAR team:

- The week index is **independent per bus** (Bus 3's "week 50" is not the same calendar week as Bus 7's "week 50"). All temporal features have to be computed per bus.
- `Avg_Qloss`, `Avg_Qcyc`, and `Avg_Qcal` are all **cumulative** — they only ever go up.

### 2.2 What it looks like after cleaning

The data-processing stage (`src/data_processing.py`) renames every column to clean snake_case, parses dates, attaches an integer `bus_id`, flags no-service weeks (`is_service_week = weekly_distance > 0`), and forward-fills cumulative columns within each bus group so that idle weeks don't reset the running total.

After cleaning we have **1,777 rows** across all 12 buses (143–154 weeks per bus), with seven no-service weeks across the fleet. The target `avg_qloss` ranges from about 0.56% to 20.09%.

### 2.3 Engineered features

Raw weekly aggregates aren't quite enough. In `src/feature_engineering.py` we compute 14 derived features that try to capture the *shape* of weekly operation, not just its scale:

- **Delta targets**: `delta_qloss`, `delta_qloss_cycling`, `delta_qloss_calendar` — week-over-week change in the cumulative loss columns. These give us a second target (the per-week increment) that is much harder to predict but much more informative if you can.
- **Operating signature**: `soc_range = max_soc - min_soc`, `regen_ratio = weekly_regen_energy / weekly_net_energy`, the time-utilization fractions `driving_frac` / `charging_frac` / `parking_frac`, and `temp_delta = avg_batt_temp - avg_amb_temp`.
- **Schedule**: `route_count` — number of distinct routes operated that week.
- **History**: 1-week lagged versions of `delta_qloss` and `avg_batt_temp`, plus 4-week rolling means of `weekly_distance` and `avg_batt_temp`.

The output `feature_matrix.csv` is what every model trains on.

## 3. Initial approaches

### 3.1 What we deliberately did *not* do

We chose, deliberately, not to throw the kitchen sink at this problem. There is a strong temptation in early ML work to fit nine different models, pick the one with the best CV score, and call it a day. That is what we *first* drafted, and we walked it back, because:

- In a small dataset (1,777 rows from 12 simulated trajectories), the best CV score among many models is often a noise-driven artifact.
- We need to *understand* what the model is doing, not just what it scores. Linear regression is interpretable. Random forest is interpretable enough (feature importances). Most of the rest just produce numbers.
- If the simple models can already hit the ceiling that the data allows, more complex models won't help; if they can't, we want to understand *why* before reaching for capacity.

So Batch 1 was deliberately small.

### 3.2 Models in Batch 1

Two models, defined in `src/modeling.py:get_model`:

- **Linear Regression** — wrapped in a `Pipeline([StandardScaler, LinearRegression])` so we can read its coefficients on standardized features. This is our baseline: it tells us how much of the target is explainable by a linear combination of features.
- **Random Forest** — `RandomForestRegressor(n_estimators=200, max_depth=20, random_state=42, n_jobs=-1)`. This is the obvious next-step model: handles non-linearity, gives feature importances, and is robust enough to be our "what if linear isn't enough" answer.

### 3.3 Targets

We model both targets:

- **Cumulative target** (`avg_qloss`) — predict total capacity loss to date. High R² achievable but dominated by the time trend.
- **Delta target** (`delta_qloss`) — predict the *weekly increment* in capacity loss. Much smaller variance; this is the target that forces the model to actually use the operating-condition features.

For the delta target, every cumulative column is dropped from the feature set to prevent the trivial leak where `cum_t - cum_{t-1}` reconstructs the target.

### 3.4 Three evaluation splits

This is where we spent a lot of design effort. A single train/test split would tell us almost nothing for a problem like this — different splits answer fundamentally different questions:

| Split | What it asks |
|---|---|
| **GroupKFold** (4 folds, grouped by `bus_id`) | "If I held out *some* buses, can the model trained on the rest predict them?" — generalization across the fleet. |
| **Leave-buses-out** (buses 10, 11, 12 held out) | A harder, fixed-holdout version of the same question — strict generalization to three never-seen buses. |
| **Temporal 80/20** (first 80% of each bus's weeks → train; last 20% → test) | "Can the model **forecast forward in time** for buses it has already seen?" — this is the question that matters for a live degradation predictor. |

Every run records all three. That's deliberate: a model can be excellent at one and terrible at another, and we want to see that contrast.

### 3.5 Run-based reproducibility

Every model fit is its own *run*: it gets a numbered folder under `results/<dataset>/runs/` containing a full config snapshot, all metrics, the predictions per row, the figures, and a markdown summary. A `run_log.csv` keeps one row per run for cross-run analysis. No result in this report is hand-typed — every number can be reproduced and re-derived.

### 3.6 The four Batch-1 runs

```
001_linear_regression_cumulative
002_linear_regression_delta
003_random_forest_cumulative
004_random_forest_delta
```

That is the full Batch-1 matrix: two models × two targets.

## 5. What those results mean

### 5.1 The good news

Cross-bus generalization is essentially solved on this simulated dataset. R² ≈ 0.99 on the cumulative target under both GroupKFold and the strict leave-buses-out split says that, given a partial trajectory of buses 1–9, the models can correctly place buses 10–12 on the same aging curve. Even on the harder *delta* target — the per-week increment, where there is no time-trend to lean on — both models score above 0.95 R² across folds.

In other words: **if we know how a bus is being driven, we can predict its weekly aging well, as long as we already have similar buses to learn from**.

### 5.2 The bad news

The temporal split tells a completely different story. R² is **negative** in three of the four runs — meaning the model performs *worse than just guessing the mean of the test set*. The picture is:

- Train the model on the first 80% of each bus's weeks (which it fits beautifully — see the 0.99 GroupKFold R²).
- Forecast the remaining 20%.
- The forecast diverges from the actual trajectory almost immediately.

You can see this visually in `results/<dataset>/figures/forecast_<model>_cumulative.png` (generated by `scripts/make_forecast_plot.py`): the train-portion fit (green dashed) hugs the actual curve, and then the forecast (red dashed) peels off as soon as the boundary is crossed.

### 5.3 Why this happens

Mechanically, both models are doing essentially the same thing: they are *interpolating* among rows that look similar to ones they've seen. Cumulative `avg_qloss` early in life looks very different from `avg_qloss` late in life — temperatures, cycle counts, distances are all different. When we train only on early-life weeks, the model has never seen a row where the cumulative cycles are 60+ or where the calendar age is three years, and its extrapolation in that regime is unreliable.

Random Forest is *especially* bad at this because trees can't extrapolate by construction — their predictions are bounded by the training-set range. For the cumulative target, that is fatal: late-life targets are simply higher than anything the trees saw at training time.

This is the canonical "interpolation easy, extrapolation hard" failure mode for tabular ML on time-series data.

### 5.4 What we are taking from Batch 1

- **The pipeline works.** Cleaning, feature engineering, modeling, evaluation, and reporting all run end-to-end and are reproducible.
- **The simulated data is highly predictable across buses** but is not, with these features, predictable into the future from a partial history.
- **The next leverage points are not "more models"** — they are (a) better-targeted features that change only in lockstep with degradation, (b) explicit time-series modeling rather than tabular regression, and (c) a richer set of simulated trajectories to train on (more on this below).

## 6. What we did last — noise injection

The clean simulated dataset is "too perfect" — every value is the simulator's noise-free output. Extremely predictable Real on-vehicle measurements will not be like that. To test how robust our pipeline is to dirty inputs, we built a parallel **noisy variant** of the dataset.

### 6.1 Design rules

1. **Same code path.** No fork of the modeling code. Activate via `DATASET_VARIANT=noisy-v1` env var; everything else is identical.
2. **Inputs only.** We perturb operating measurements (SOC, temperatures, energies, distances, times, cycles). We do **not** perturb the target — we want to know whether models built on noisy inputs can still predict the clean truth.
3. **Per-column noise specs.** `COLUMN_NOISE` in `src/config.py` defines, per column, an additive (`x + N(0, σ)`) or relative (`x · (1 + N(0, σ))`) noise rule with optional clip bounds. Standard deviations were chosen to be physically plausible (e.g. battery temp σ=0.2 °C, ambient σ=0.75 °C, distances 3% relative).
4. **Consistency repairs.** After perturbation we run three repairs so the noisy dataset is still self-consistent:
   - SOC ordering is restored (`min ≤ avg ≤ max`).
   - The weekly time budget is renormalized so driving + charging + parking = the original total for that week (noise reshuffles how the week was spent, not how long it was).
   - Cumulative totals (`weekly_tot_distance`, etc.) are rebuilt as cumsums of the noised weekly values, so weekly and cumulative columns can't disagree.
5. **Service weeks only**, and a fixed seed (42) so the noisy dataset is reproducible byte-for-byte.

### 6.2 Outputs

The noisy run is in `results/mar-2026-noisy-v1/`. It includes:

- The same four Batch-1 runs (LR/RF × cumulative/delta).
- `comparison_to_clean.csv` — a metric-by-metric table of `noisy − clean` deltas.
- `comparison_summary.md` — a short markdown table of R² and RMSE deltas per (model, target).
- `implementation_summary.md` — the slide-friendly writeup.

### 6.3 What the comparison shows (high level)

The deltas are small. Across all four runs:

- GroupKFold and holdout R² move by roughly 10⁻³ — within noise of itself.
- Temporal R² is also slightly more negative on the noisy variant (it was already negative).

That tells us our models are **robust to plausible measurement noise on the inputs**: the cross-bus generalization story doesn't break, and the temporal-forecasting failure isn't being caused by clean-vs-noisy inputs — it is structural to the modeling choice.

(For the full numbers, see `results/mar-2026-noisy-v1/comparison_to_clean.csv`. Don't quote them from memory in slides; they're auto-regenerated whenever the noisy run is re-executed.)

## 7. Next steps

### 7.1 Vary the simulation parameters (immediate next experiment)

Adding sensor noise to the existing simulation only varies measurement uncertainty — the underlying physics of the simulated buses is identical to the clean run. The next experiment, requested by the CAR team, is to **vary the simulation inputs themselves** so we get a richer set of trajectories.

Concretely, the CAR team can re-run the simulator with different choices for things like:

- driving profiles (route mix, stop frequency, average speed),
- climate (more aggressive ambient-temperature swings, hot-summer / cold-winter regimes),
- charging strategy (slower vs faster charging, different SOC targets),
- duty-cycle intensity (more or fewer weekly cycles).

Each new sweep lands as a new `BASE_DATASET=<sweep-name>` under `data/raw/`. The pipeline does not change. We then ask:

- Do the models trained on one parameter regime generalize to another? (This is the analog of the GroupKFold question, but across regimes instead of buses.)
- Does the temporal-forecast failure go away if we have *much more* trajectory diversity to train on? Or is it a structural issue with tabular regression?

This experiment is the most likely to move the needle, because the bottleneck right now is trajectory diversity, not model capacity.

### 7.2 Properly time-series modeling

Even with more data, predicting "what does the cumulative qloss look like 30 weeks from now?" is a forecasting problem, not a regression problem. Reasonable next directions:

- Predict residuals on top of an explicit aging model rather than predicting the raw curve.
- A small recurrent or state-space model that consumes the per-bus history rather than a flat feature vector.
- A monotonic constraint — `avg_qloss` cannot decrease — wired into the model class.

### 7.3 Bridge to real-world data

The medium-term goal is to point the same pipeline at on-vehicle data from the COTA fleet. To make that handoff smooth:

- Schema mapping — define the rename from real-telemetry columns to our snake_case names so `src/config.py:COLUMN_RENAME` is the only thing that needs to change.
- Data-quality gates — extend the noise/consistency-repair logic with checks that real data will need (sensor dropouts, clock skew, missing weeks).
- A "validation week" protocol — what does it look like to retrospectively score a forecast made *N* weeks ago against what actually happened?

### 7.4 Live degradation prediction (long-term)

The end-state vision is a system that, for each bus in the fleet, continuously updates a degradation forecast as new operating data comes in. At that point the pipeline runs forward — every new week's data ingests, features rebuild, the model retrains or warm-starts, and the forecast for the next *N* weeks updates. The infrastructure we have built — environment-switchable datasets, run-based reproducibility, side-by-side variant comparison — is the foundation that makes that step a matter of plugging in new data, not rewriting the project.
